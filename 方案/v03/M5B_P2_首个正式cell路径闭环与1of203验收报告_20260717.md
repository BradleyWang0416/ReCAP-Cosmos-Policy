## 3. DCP 审计修正

路径闭环后，原 v5 validator 暴露第二个独立问题：它要求每个 DCP rank
文件都非空，但 PyTorch DCP 对 replicated scheduler/trainer state 只让
rank 0 持有 payload，其余 rank 文件是合法的 0-byte 占位文件。

该布局在 step 1000–7000 七个 checkpoint 上完全一致：

- model metadata 引用 rank 0–3；
- optim metadata 引用 rank 0–3；
- scheduler metadata 只引用 rank 0；
- trainer metadata 只引用 rank 0。

修正后的审计器直接使用 PyTorch `FileSystemReader` 解析 `.metadata`，并
执行以下 hard-fail 检查：

1. model/optim/scheduler/trainer 四组件必须存在；
2. 每组件必须存在四个 rank 文件；
3. `.metadata` 必须非空且可由 PyTorch 解析；
4. 每个 metadata storage reference 必须解析到预期 rank 文件；
5. 每个 offset/length 必须落在实际文件范围内；
6. 被 metadata 引用的空文件禁止通过；
7. 只有未被 metadata 引用的空占位文件允许通过。

修正冻结为：

- `M5B_P2_dcp_audit_closure_v1.json`；
- `M5B_P2_dcp_audit_closure_v1.lock.json`；
- `scripts/human2robot_m5b_p2_dcp_audit_closure.py`。

Docker 回归结果：

- 修复前：1 failed，准确命中 scheduler 空占位文件误拒；
- 修复后：2 passed；
- 截断被引用 payload 的负向测试仍按预期 hard fail。

该审计修正位于训练 source snapshot 范围之外。Docker 内重新计算的受控
source SHA256 仍为：

`07fe5ee3a9b90037ed724a94da76d9d87476e6eaea554af8c3ddb420ff3da511`

因此数据、模型、优化器和训练代码没有改变，v5 activation 继续有效。
